Dependency Ordering Best Practices

Foundation Layer (no dependencies):
  - VPC, Subnets, Route Tables

Security Layer (depends on Foundation):
  - Security Groups, NACLs, IAM Roles

Data Layer (depends on Foundation + Security):
  - RDS, DynamoDB, S3, ElastiCache

Compute Layer (depends on all above):
  - ECS, Lambda, EC2

Application Layer (depends on Compute):
  - DNS, CDN, Monitoring
Component 3: Drift Detection
This is the key to drift detection:

terraform plan -detailed-exitcode
Exit Code 0: Success, no changes needed (no drift)
Exit Code 1: Error occurred (configuration issue)
Exit Code 2: Success, changes detected (DRIFT!)

Most people think any non-zero exit code is an error. But exit code 2 is success — it successfully detected changes!

Implementation Example (in a CI script):
START=$(date +%s)
terraform apply -auto-approve -lock-timeout=600s
END=$(date +%s)
DURATION=$((END - START))
echo "Apply took ${DURATION} seconds"
if [ $DURATION -gt 300 ]; then
  echo "WARNING: Apply took over 5 minutes, possible lock contention"
fi

 MAX_RETRIES=5
    RETRY_DELAY=30

    for i in $(seq 1 $MAX_RETRIES); do
      echo "Attempt $i of $MAX_RETRIES"

      if terraform plan -no-color -out=tfplan 2>&1; then
        echo "Plan succeeded"
        exit 0
      fi

      # Check if the failure was due to a lock
      if terraform plan -no-color -out=tfplan 2>&1 | grep -q "Error acquiring the state lock"; then
        echo "State is locked. Waiting ${RETRY_DELAY}s before retry..."
        sleep $RETRY_DELAY
        RETRY_DELAY=$((RETRY_DELAY * 2))  # Exponential backoff
      else
        echo "Plan failed for a non-lock reason"
        exit 1
      fi
    done

    echo "Failed to acquire lock after $MAX_RETRIES attempts"
    exit 1

#!/bin/bash
# Script to target only changed resources
set -e

# Generate plan and store output
PLAN_OUT="tf.plan"
terraform plan -detailed-exitcode -out=${PLAN_OUT} -no-color | tee plan.txt

# Parse plan for changes, creating -target arguments
RESOURCES=$(grep -E '^\s*([~+-]|[-+]{2})' plan.txt | \
            grep -vE '\(\s*known\s+after\s+apply\s*\)' | \
            awk '{print $2}' | \
            sed 's/^[[:space:]]*//' | \
            sort -u | \
            sed 's/^/-target=/')

# Apply changes if any exist
if [ -z "$RESOURCES" ]; then
    echo "No changes detected, skipping apply."
else
    echo "Applying changes for: $RESOURCES"
    terraform apply -auto-approve $RESOURCES
fi

# Cleanup
rm -f ${PLAN_OUT} plan.txt

# configs/project-a.env

#!/bin/bash

# Exit immediately if a command exits with a non-zero status.
set -e

# Define the directory containing your Terraform files (default to current directory)
TERRAFORM_DIR=${1:-.}

cd "$TERRAFORM_DIR" || { echo "Error: Directory $TERRAFORM_DIR not found."; exit 1; }

echo ":: Running terraform init..."
# Initialize the directory without a backend for speed, as we only need validation.
terraform init -backend=false -input=false

echo ":: Checking Terraform formatting..."
# Check if files are formatted correctly and show differences if not.
# The --check flag makes the command return a non-zero exit code if files need formatting.
if ! terraform fmt -check -recursive -diff; then
    echo ""
    echo "Error: The following Terraform files are not properly formatted."
    echo "Run 'terraform fmt -recursive' in the $TERRAFORM_DIR directory to fix."
    exit 1
fi
echo "All Terraform files are properly formatted."

echo ":: Running terraform validate..."
# Validate the configuration for syntax and internal consistency.
terraform validate

echo "Success: Terraform formatting and validation passed."
